1

O desafio

  • Sistema legado que atualiza a qualidade dos itens de um estoque, todos os dias.
  • Funciona, mas é difícil de manter, testar e estender.
  • Missão: modernizar preservando o comportamento existente e adicionar a categoria Conjurados.
  • Restrição: a classe Item e a propriedade items nunca podem ser alteradas.
2

Diagnóstico técnico

Condicionais aninhados

até 4 níveis em um único bloco

Identificação por texto

nome do item comparado repetidas vezes

Responsabilidades misturadas

um método decide tudo, para todo tipo de item

Baixa testabilidade

impossível testar uma regra isoladamente

Bug confirmado: o item Conjurado já existia na simulação, mas degradava como item comum.

Um arquivo com tudo misturado deu lugar a uma pasta por responsabilidade: regra de negócio em domain, orquestração em application.

Antes: um único método para tudo

legacy/gilded_rose.py
def update_quality(self): for item in self.items: if item.name != "Aged Brie" and item.name != "Backstage passes...": if item.quality > 0: if item.name != "Sulfuras, Hand of Ragnaros": item.quality = item.quality - 1 else: if item.quality < 50: item.quality = item.quality + 1 if item.name == "Backstage passes...": if item.sell_in < 11: if item.quality < 50: item.quality = item.quality + 1 if item.sell_in < 6: if item.quality < 50: item.quality = item.quality + 1 if item.name != "Sulfuras, Hand of Ragnaros": item.sell_in = item.sell_in - 1 if item.sell_in < 0: if item.name != "Aged Brie": if item.name != "Backstage passes...": if item.quality > 0: if item.name != "Sulfuras, Hand of Ragnaros": item.quality = item.quality - 1 else: item.quality = item.quality - item.quality else: if item.quality < 50: item.quality = item.quality + 1

29 linhas, 4 níveis de aninhamento, 4 tipos de item decididos no mesmo bloco. Nenhuma regra para o Conjurado.

Cada regra cabe em poucas linhas, sem aninhamento, e pode ser testada sozinha.

6

Arquitetura: antes e depois

antes

Uma única classe (GildedRose) concentra a orquestração e a regra de negócio de todos os tipos de item ao mesmo tempo.

depois

domain guarda o Item (intocado) e uma estratégia por tipo de item. application só orquestra: identifica o tipo e delega ao domínio.

Clean Architecture simples: separa a regra de negócio da orquestração, sem alterar Item nem items.

9

Como o processo foi conduzido

  1. Diagnóstico do código legado, com testes ainda ausentes.
  2. Documentação de requisitos e plano de contingência.
  3. Testes de caracterização antes de qualquer mudança.
  4. Refatoração incremental, sempre com os testes passando.
  5. Implementação da regra dos itens Conjurados.
  6. Evidências finais e comparação antes e depois.
10

Lições aprendidas

  • Ler o código não basta. Foi necessário executar o sistema para confirmar o comportamento real antes de mudar qualquer coisa.
  • Testes de caracterização não substituem entender a regra de negócio, eles só documentam o que já existe.
  • Refatorar em passos pequenos custa mais tempo, mas elimina o risco de uma mudança grande esconder uma regressão.
11

Resultado

Sistema modernizado preservando o comportamento original, com a regra de itens Conjurados implementada e validada por testes, e uma arquitetura que facilita a inclusão de novas categorias no futuro.
